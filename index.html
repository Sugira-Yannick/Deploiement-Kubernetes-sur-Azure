1. Introduction

Ce rapport détaille la mise en œuvre d'une application multi-conteneurs sur Azure Kubernetes Service (AKS). L’objectif était de déployer une architecture trois tiers — frontend Nginx, backend Node.js, base PostgreSQL —, exposer le frontend publiquement, garantir la communication interne et prouver la scalabilité horizontale.

Le cluster AKS a été provisionné avec un seul nœud (Standard_D2s_v3) dans le groupe de ressources GR-9. Tous les composants ont été orchestrés via des manifests YAML, en contournant intelligemment les restrictions d’Azure (absence d’ACR, limitations de VM).

2. Kubernetes — Orchestrateur de conteneurs

Kubernetes (K8s) automatise le déploiement, la mise à l’échelle et la gestion d’applications conteneurisées. Il agit comme un chef d’orchestre : tolérance de panne, équilibrage de charge, stockage persistant, mises à jour sans interruption. Dans ce TP, K8s a coordonné les trois services de manière isolée et reproductible.

🔁 Valeur ajoutée : Redémarrage automatique des Pods défaillants, résolution de service interne via DNS, scaling instantané.

3. Docker vs Kubernetes : différence fondamentale

Docker construit et exécute des conteneurs unitaires ; Kubernetes orchestre des armadas de conteneurs sur plusieurs machines. Ici, Docker a servi à empaqueter la logique (images Node, Nginx, Postgres) tandis que K8s les a déployés, liés et supervisés dans le cloud Azure.

4. Pods, Deployments, Services — piliers K8s

  • Pods : unité atomique, chaque composant (frontend, backend, postgres) a son propre Pod.
  • Deployments : gestion des réplicas, mises à jour, stratégie de rollout. 3 Deployments ont été définis.
  • Services : exposition réseau stable. Le frontend-service de type LoadBalancer a obtenu une IP publique. backend-service et postgres-service en ClusterIP assurent les communications internes.

5. Architecture applicative

Utilisateur → LoadBalancer (IP publique) → Frontend (Nginx) → Backend (Node.js) → PostgreSQL (avec PVC)

Le backend expose une API REST (/api/health, /api/data) qui incrémente un compteur de visites en base. Le frontend consomme cette API et affiche dynamiquement le résultat.

6. Environnement & contraintes techniques

  • ❌ Docker Desktop indisponible (virtualisation désactivée)
  • ❌ Registre ACR interdit par la souscription Azure
  • ❌ SKU Standard_B2s non autorisé → bascule sur Standard_D2s_v3
  • ✅ Solution : ConfigMaps pour injecter le code backend/frontend, images publiques officielles, cluster éphémère

7. Provisioning du cluster AKS

az aks create --resource-group GR-9 --name aks-gr9-tp --location centralus \
  --node-count 1 --node-vm-size Standard_D2s_v3 --enable-managed-identity --generate-ssh-keys

Le cluster a été opérationnel en ~8 minutes. Contexte récupéré via az aks get-credentials puis validation avec kubectl get nodes (nœud Ready).

8. Déploiement Kubernetes complet

Les manifests ont été appliqués dans l’ordre : namespace → secret PostgreSQL → déploiement & service Postgres → backend (avec ConfigMap contenant server.js) → frontend Nginx + ConfigMap nginx.conf.

kubectl apply -f k8s/00-namespace.yaml
kubectl apply -f k8s/01-secret.yaml
kubectl apply -f k8s/02-postgres.yaml
kubectl apply -f k8s/03-backend.yaml
kubectl apply -f k8s/04-frontend.yaml

10. Scalabilité — montée en charge du backend

Pour simuler un pic de trafic, le deployment backend a été passé de 1 à 3 réplicas :

kubectl scale deployment backend --replicas=3 -n tp-k8s

Kubernetes a instantanément créé deux Pods supplémentaires. L'équilibrage de charge est automatique via le service ClusterIP. Cette démonstration confirme l'élasticité native de K8s.

11. Résolution d’incidents & adaptations

  • CrashLoopBackOff PostgreSQL : erreur "lost+found directory" → ajout de la variable PGDATA: /var/lib/postgresql/data/pgdata pour isoler le répertoire de données.
  • Backend en échec : dossier /app manquant → commande mkdir -p /app dans le script init du conteneur.
  • ACR bloqué : contournement par ConfigMap (code source injecté directement) + images publiques stables.
  • VM SKU refusé : adoption de Standard_D2s_v3 après vérification des offres régionales.
💡 Ces corrections ont permis d’obtenir un environnement entièrement fonctionnel, sans dépendance à des services Azure supplémentaires.

12. Conclusion & livrables

Le TP a atteint tous les objectifs : cluster AKS opérationnel, application trois tiers déployée, exposition publique, communications inter-services validées, scaling horizontal réussi. L'approche “ConfigMap + images publiques” a prouvé sa robustesse face aux restrictions cloud.

Ce projet illustre concrètement la puissance de Kubernetes : reproductibilité, auto-réparation, abstraction de l’infrastructure. L’architecture peut être étendue avec un ingress controller, des sondes de santé, et un stockage managé Azure.

# Connexion & cluster
az login
az aks create -g GR-9 -n aks-gr9-tp --node-count 1 --node-vm-size Standard_D2s_v3
az aks get-credentials -g GR-9 -n aks-gr9-tp

# Déploiement
kubectl apply -f k8s/00-namespace.yaml
kubectl apply -f k8s/01-secret.yaml
kubectl apply -f k8s/02-postgres.yaml
kubectl apply -f k8s/03-backend.yaml
kubectl apply -f k8s/04-frontend.yaml

# Vérifications
kubectl get all -n tp-k8s
kubectl get svc frontend-service -n tp-k8s

# Scaling
kubectl scale deployment backend --replicas=3 -n tp-k8s

# Nettoyage
az aks delete -g GR-9 -n aks-gr9-tp --yes
